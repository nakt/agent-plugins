# draft-service-brief

新しい取り組みをサービス化した際の「サービス紹介の元原稿」を、NotebookLM や Claude Design などのデザイン変換ツールにそのまま投入できる構造化文書としてドラフトするスキル。散らばったアイデアメモ・実装ドキュメント・伝えたい要点を寄せ集めて 1 本の文章にまとめる。品質の要は、対象顧客と課題、サービスの中身、アウトカム(顧客にとって何が変わるか)、差別化要因が、それぞれ独立した読み物として成立していること。デザインは下流ツールに任せ、こちらは意味を文字にすることに集中する。

## モード判定

- 改訂モード: このセッションで直前に生成した原稿がある、またはユーザーが既存の原稿ファイルを明示的にパスで指して修正・TBD の穴埋めを依頼した場合。会話履歴の自動検出はしない(誤爆防止)。改訂モードでは Phase 1〜3 をスキップし、既存原稿を Read → 該当章のみ Edit → Phase 5(セルフレビュー 8 観点、省略しない)→ Phase 6 を実行する
- 新規モード: それ以外。Phase 1 から順に進める

## 全体フロー

### Phase 1: 元ネタ受領と出力先確認

1. 引数にパスがあれば Read する(複数指定可)。会話中に元ネタが提示されていればそれも使う。何もなければ「元ネタが手元にあるか、あればどこか」を平文で確認する。ゼロからでも進行できる
2. 元ネタからヒアリングチェックリストの各項目に転記できる情報を抽出し、「読み取れたこと / 不足していること」を一覧で提示する(解釈ずれの早期訂正のため)
3. サービス名(仮称可)を確認する
4. 出力先を確認する。未指定ならカレントプロジェクトの `.workspace/NN_service-brief-{サービス名スラッグ}/brief.md` を提案する(NN は既存の最大値 + 1、`.workspace/` がなければ新規作成)。既存ファイルへの無断上書きはしない

### Phase 2: ヒアリング

ヒアリングチェックリストの必須項目を聞き切る。元ネタから読み取れた項目は質問せず、「元ネタから◯◯と理解した」と要約提示して確認のみ行う。

質問方式の振り分け:

- AskUserQuestion を使う: 回答が事前に列挙できる選択式の質問(想定読者の役職層、オプション章の取捨、章立て案の承認、サービス型の確定)
- 平文で聞く: 自由記述が本質の質問(サービス内容、課題の具体像、プロセス/アプローチ/機能の中身、アウトカムの具体像、差別化の根拠)。1 回のメッセージに最大 5 問、番号付きでまとめ、「精度の高い原稿のために確認したい」と趣旨を添える

必須項目にユーザーが「わからない」と答えた場合は仮案を提示して合意を取る(空欄で進めない)。任意項目が不明な場合は即 TBD 化して進む。

とくにアウトカムは「レポートを納品します」と答えられがちなので、「その結果、顧客の業務・意思決定・数値・リスクが具体的にどう変わるか」を必ず一段深掘りする(それは出力であってアウトカムではない、と明示的に切り返してよい)。4 側面(業務・意思決定・数値・リスク)は書き手向けのチェックリストで、最低 1 側面が具体的に埋まれば OK。埋まらない側面は省略可(TBD 化はしない)。

### Phase 3: 章立て合意

1. [references/section-guide.md](references/section-guide.md) を読む
2. ヒアリング結果から「サービスの中身」章のサービス型を判定する。トップ 2 候補(理由付き)を提示し、AskUserQuestion で 5 択(データ預かり型 / コンサル・受託型 / プロダクト・ツール型 / 研修・教育型 / その他)から確定する
3. 確定したサービス型のサブ構造を組み込んだ章立て案を「番号 / 章名 / 1 行主張案」の表で提示する。オプション章の追加提案条件(section-guide.md)に該当するものがあれば併せて提案(採否は AskUserQuestion multiSelect)
4. AskUserQuestion で承認/調整を得る。調整要望への再提案は最大 3 回まで。3 回目に至ったら AskUserQuestion で「現行案で進める / 中断」の 2 択を提示し、進める選択なら未解決点は TBD 化する

### Phase 4: 原稿生成

1. [references/output-template.md](references/output-template.md) を読む
2. 合意した章立てで各章を書き切って出力先に Write する。各章は箇条書きに逃げず、まず 1〜2 段落の文章で骨子を書く。分量の上限は設けない(元ネタ・ヒアリングから引き出せた内容は削らずに書き切る。ただし冗長・重複・根拠のない水増しはしない)
3. ユーザーから得ていない数値・実績・顧客名は書かず、`TBD: ※◯◯を確認` 形式で埋めて末尾の TBD 一覧に登録する(本文と一覧の 1:1 対応が必須)

### Phase 5: セルフレビュー

section-guide.md のセルフレビュー観点 8 項目をドラフト全体に対して点検し、問題があれば Edit で修正する。修正後、「観点 / 判定 / 修正内容」の一覧をユーザーに報告する。

### Phase 6: TBD 一覧と次アクション

TBD 一覧を会話にも再掲し、次のアクションを提示して終える:

- 生成した Markdown はそのまま NotebookLM や Claude Design 等のデザイン変換ツールに投入できる
- TBD が確定したら改訂モードで反映できる
- pptx や Marp スライドそのものの生成は本スキルの範囲外

## ヒアリングチェックリスト

必須(聞き切る。「わからない」なら仮案提示 → 合意で進める):

1. サービス名(仮称可)と、ひとことで言うと何のサービスか
2. 対象顧客: 業種・部門・役職・規模。原稿を最終的に読む人は誰か(意思決定者か現場か)
3. 顧客の課題: 何にどう困っていて、現状どう対処しているか(代替手段の限界)
4. サービスの中身: サービス型(データ預かり / コンサル・受託 / プロダクト・ツール / 研修・教育 / その他)の判定に必要な情報。何を扱い、どう関わり、何を返すか
5. アウトカム: 業務・意思決定・数値・リスクの 4 側面のうち最低 1 側面以上、具体的にどう変わるか(納品物ではなく「その結果」)
6. 差別化要因: 代替手段(他社・内製・現状維持)と比べた強み 2〜3 点(理由付き)

任意(不明なら TBD 化):

- 背景・なぜ今このサービスか、なぜ我々がやるのか
- 実績・効果数値
- 料金・契約形態
- 提供体制・標準期間
- データの取り扱い・セキュリティ(預かり型サービスでは確認推奨)
- 進め方(契約〜納品の流れ)

## Key Principles

- アウトカムを納品物と混同しない: 納品物は「何を返すか」、アウトカムは「その結果、顧客の業務・意思決定・数値・リスクが何にどう変わるか」。この分離をヒアリングでもレビューでも徹底する
- 差別化は中身章に埋め込む: プロセス / アプローチ / 機能 / プログラム等の中身章は単なる手順・機能の説明ではなく、他社・内製と違うポイントを織り込んで書く。差別化要因章とも重ねてよい
- スライドではなく文章: 箇条書きの羅列に逃げず、まず文章で骨子を書く。デザイン変換ツールはセクションの意味を掴んで自分でデザインするので、こちらは意味を明確に文字にすることに集中する
- 顧客視点の言葉で書く: 自社機能の自慢ではなく、顧客の課題・利益の言葉に翻訳する
- 数値・実績・顧客名を捏造しない: ユーザーから得ていない値はすべて TBD にする
- 組織固有の定型(会社紹介・結びページ等)は含めない
- pptx / Marp スライドの生成はしない(元原稿の Markdown まで。デザイン化は NotebookLM / Claude Design 等の下流ツールに委ねる)
- markdownlint は PostToolUse hook が自動実行するため自分で実行しない。エラーのフィードバックを受けたら Edit で修正する
