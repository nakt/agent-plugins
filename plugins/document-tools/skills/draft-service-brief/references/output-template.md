## 記入ルール

- 各章はまず 1〜2 段落の文章で骨子を書く。箇条書きの羅列で埋めない。NotebookLM / Claude Design はセクションの意味を掴んで自分でデザインするので、こちらは意味を文字にすることに集中する
- 必要な場合は短い箇条書きを段落の後に添える(3 点以内目安)
- 未確定情報は本文中に `TBD: ※◯◯を確認` 形式で埋め込み、必ず末尾の TBD 一覧にも同じ項目を載せる(1:1 対応)。ユーザーから得ていない数値・実績・顧客名は推測で書かない
- サービスの中身章のサブ構造は合意したアーキタイプのテンプレートに従う(section-guide.md 参照)。「その他」型を選んだ場合は H3 の項目名も自由に決めてよい
- オプション章を追加する場合は差別化要因の後・TBD 一覧の前に挿入する。順序は「背景 / データの取り扱い / 提供体制 / 進め方 / 事例 / 料金」を目安とする(重要度・読み手の関心順)
- 分量の上限は設けない。元ネタ・ヒアリングから引き出せた内容は削らずに書き切る。ただし冗長・重複・根拠のない水増しはしない(Phase 5 のセルフレビュー観点 4・5 で弾く)
- メタ情報表の「サービス型」は判定結果を明記する(下流ツール投入時のヒントにもなる)

## 記入例

コールセンター問い合わせデータの LLM 分析・改善提案サービスを例にした記入例。データ預かり型のサブ構造を採用したケースで、サマリ〜アウトカムの主要部分を示す。

````markdown
## サマリ

コールセンター問い合わせ改善パートナー(仮称)は、BtoC 事業者のコールセンター運営責任者を対象に、CRM に蓄積された問い合わせログを預かって LLM で全件分析し、傾向・根本原因・改善アクション案をまとめたレポートとしてお返しするサービスです。手作業のタグ付けに依存してきた分析工程を自動化することで、これまで場当たりだった FAQ 整備やオペレーション改善の意思決定を、毎月データに裏付けられた形で回せるようにします。

## 対象顧客と課題

対象は、月数千〜数万件の問い合わせを受けている BtoC 事業者のコールセンター運営責任者です。問い合わせログは CRM に蓄積されていますが、傾向分析はオペレーターがつけたタグの集計に頼っており、タグの粒度も付与率もまちまちで、FAQ 整備の意思決定材料として使えていません。現場は「重要そうな案件」の記憶ベースで手を打っており、施策の再現性も検証もされていないのが実情です。

## サービスの中身

### 入力: お預かりするもの

コールセンターの問い合わせログ(TBD: ※対応可能な受領形式・粒度を確認)。オペレーターが付与した既存タグがあれば併せてお預かりし、分析の参考にします。

### プロセス: 分析・作業の中身

お預かりしたログは、まず問い合わせ内容を LLM で読解し、事前定義したカテゴリではなく、そのデータから自然に浮かぶ分類軸を提案します。この点が既製の分析ツールと最も異なる部分で、業界固有・企業固有の分類軸を強制せず、実データから浮かび上がる軸を採用することで、想定外の課題領域(例: 特定キャンペーン後の解約意向)を取りこぼしません。分類後、頻度・時系列変化・根本原因を LLM に構造化させ、改善アクション候補を優先度付きでまとめます。

### 出力: 納品するもの

問い合わせ傾向レポート(TBD: ※標準ページ数を確認)と、担当者向けの報告会 1 回。レポートには、分類ごとの件数・時系列変化、根本原因の推定、改善アクション候補(優先度・想定効果・実行難度付き)を掲載します。

## アウトカム

これまで四半期に一度、経験ベースでまとめていた問い合わせ傾向レポートが、毎月データに裏付けられた形で手元に届くようになります。FAQ 整備の優先順位付けが体感ではなく件数・影響度で議論できるようになり、施策の効果検証も次月のデータで自然に回せる状態に変わります。オペレーターがタグ付けに費やしていた工数も削減できます(TBD: ※削減工数の目安を確認)。
````

このように TBD を本文に埋めた場合は、末尾の TBD 一覧に「入力 / 対応可能な受領形式・粒度 / サービス責任者」等の行を必ず追加する。
