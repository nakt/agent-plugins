# 章立てガイド

draft-service-brief の Phase 3(章立て合意)の直前に読む。必須 5 章の書き方、サービスの中身章のアーキタイプ別サブ構造、オプション章の追加提案条件、生成後のセルフレビュー観点を定義する。詳細な参照例は「データ預かり型」のみに置き、他アーキタイプは骨子と 1〜2 行の例に留める(SKILL 全体を肥大化させないため)。

## 必須 5 章の書き方

### 1. サマリ

- 役割: 忙しい読み手が 1 段落だけ読んでもサービスの正体が掴めるようにする
- 書き方: サービス名 + 「誰の / 何を / どう解いて / その結果何が変わるか」を 1 段落(3〜5 文)。抽象的な形容詞を避け、動詞と名詞で書く。この段落は下流ツールでもトップページ・イントロダクションに使われる可能性が高い、と意識する
- 良い例: 「コールセンター問い合わせ改善パートナーは、BtoC 事業者のコールセンター運営責任者を対象に、CRM に蓄積された問い合わせログを預かって LLM で全件分析し、傾向・根本原因・改善アクション案をまとめたレポートとしてお返しするサービスです。手作業のタグ付けに依存してきた分析工程を自動化することで、これまで場当たりだった FAQ 整備やオペレーション改善の意思決定を、毎月データに裏付けられた形で回せるようにします。」
- ありがちな失敗: 機能を並べただけで、誰の何が変わるかが見えない。「AI で分析」など手段だけ書いて成果が抜ける

### 2. 対象顧客と課題

- 役割: 読み手に「これはうちの話だ」と思わせる
- 書き方: (a) 対象顧客のセグメント(業種・部門・役職・規模)を具体的に、(b) その顧客が今どう困っているか、(c) 現状どう対処していて何が限界か、を段落で書く。抽象論(「DX が求められる時代」等)から入らない
- 良い例: 「対象は、月数千〜数万件の問い合わせを受けている BtoC 事業者のコールセンター運営責任者です。問い合わせログは CRM に蓄積されていますが、傾向分析はオペレーターがつけたタグの集計に頼っており、タグの粒度も付与率もまちまちで、FAQ 整備の意思決定材料として使えていません。現場は『重要そうな案件』の記憶ベースで手を打っており、施策の再現性も検証もされていないのが実情です。」
- ありがちな失敗: 一般論から始める。自社サービスに都合よく寄せた課題設定になっている

### 3. サービスの中身

- 役割: サービスの実体を示す。差別化の核はこの章に埋め込む(差別化要因章と重ねて書いてよい)
- 書き方: サービス型に応じて後述のサブ構造テンプレートを使う。単なる手順や機能の列挙にせず、なぜそうするのか / 他社・内製と何が違うかを織り込む
- サブ構造は次節を参照

### 4. アウトカム

- 役割: 納品物ではなく「その結果、顧客の業務・意思決定・数値・リスクが何にどう変わるか」を書く。この章の質がサービス紹介文書の価値を決める
- 書き方: 業務 / 意思決定 / 数値 / リスクの 4 側面から、書けるものを段落で描写する。最低 1 側面は必ず書く(埋まらない側面は省略可、TBD 化はしない)。可能なら Before/After の対比で書く。数値は根拠がある場合のみ入れ、なければ TBD 化
- 良い例: 「これまで四半期に一度、経験ベースでまとめていた問い合わせ傾向レポートが、毎月データに裏付けられた形で手元に届くようになります。FAQ 整備の優先順位付けが体感ではなく件数・影響度で議論できるようになり、施策の効果検証も次月のデータで自然に回せる状態に変わります。TBD: ※効果数値(削減工数・改善率)を確認」
- ありがちな失敗: 「レポートを納品します」で止まる(それは出力)。「業務効率化を実現」といった抽象語で逃げる。全側面を無理に埋めて TBD だらけになる

### 5. 差別化要因

- 役割: 代替手段(他社サービス / 内製 / 現状維持)と比べたときの選ばれる理由を示す
- 書き方: 強み 2〜3 点。各点を「事実 + 代替と比べたときの意味」のセットで書く。単なる形容詞(「高精度」「安心」)で終わらせない
- 良い例: 「第一に、事前定義された分類テンプレートを使わず、顧客データから分類軸そのものを提案する点。汎用の CX 分析ツールが業種標準の分類を押し付けがちなのに対し、業界固有・企業固有の観点を取り逃がさない設計になっています。第二に、分析だけでなく改善アクション候補まで踏み込む点。TBD: ※3 点目の候補を確認」
- ありがちな失敗: 強みを 5 点も 6 点も並べて印象が薄まる。「業界最速」「業界唯一」など検証されていない主張

## サービスの中身 - アーキタイプ別サブ構造

Phase 3 でヒアリング結果からトップ 2 候補を提示し、AskUserQuestion で以下 5 択(4 型 + その他)から確定する。判定材料は次項の「典型入力パターン」を参照する。

### データ預かり型: 入力 / プロセス / 出力

顧客からデータや素材を預かり、分析・変換・整備して返すサービス。典型入力パターン: 「◯◯ログを預かって」「データを分析して」「◯◯を LLM で処理して」等の語彙。データ受領形式・粒度への言及があるかも判定材料。

サブ構造:

```markdown
### 入力: お預かりするもの
{何を、どんな粒度・形式で預かるか}

### プロセス: 分析・作業の中身
{手順の骨子。なぜその手順なのか、他社・内製と何が違うかを織り込む。差別化の核はここ}

### 出力: 納品するもの
{何を、どんな粒度で、いつ返すか}
```

参照例(プロセス部分): 「お預かりしたログは、まず問い合わせ内容を LLM で読解し、事前定義したカテゴリではなく、そのデータから自然に浮かぶ分類軸を提案します。この点が既製の分析ツールと最も異なる部分で、業界固有・企業固有の分類軸を強制せず、実データから浮かび上がる軸を採用することで、想定外の課題領域(例: 特定キャンペーン後の解約意向)を取りこぼしません。分類後、頻度・時系列変化・根本原因を LLM に構造化させ、改善アクション候補を優先度付きでまとめます。」

### コンサル・受託型: アプローチ / 進め方 / 成果物

顧客の課題に対して伴走・支援し、期間を区切って成果を出すサービス(戦略コンサル・PMO・受託開発を含む)。典型入力パターン: 「◯ヶ月伴走して」「支援する」「一緒に◯◯を設計する」「受託する」等の語彙。期間・体制への言及があるかも判定材料。

サブ構造:

```markdown
### アプローチ: 課題への向き合い方
{どんな観点・方法論で課題に向き合うか。差別化の核はここに埋める}

### 進め方: フェーズと期間
{フェーズ分けと各フェーズの活動、標準的な期間感}

### 成果物: お渡しするもの
{中間成果物・最終成果物・引き継ぎ物}
```

例(アプローチ骨子): 「現場ヒアリング → ボトルネックの定量化 → 打ち手仮説の設計 → 実行支援、を 3 ヶ月 1 サイクルで回す。初月に必ず現場に入り、机上の分析だけで進めない点が受託型コンサルとの違い。」

### プロダクト・ツール型: 機能 / 使い方 / 提供範囲

顧客が自分で使うプロダクトやツールを提供するサービス(SaaS・ライセンス販売等)。典型入力パターン: 「ツールを提供する」「SaaS」「機能を使ってもらう」等の語彙。ライセンス・アカウント・利用シーンへの言及があるかも判定材料。

サブ構造:

```markdown
### 機能: 何ができるか
{主要機能を顧客の利益ベースで。機能名の羅列でなく「これで何が解ける」の視点で書く}

### 使い方: 現場での利用イメージ
{典型的な利用シーンと顧客の作業ステップ}

### 提供範囲: サポート・データ・環境
{どこまでが提供範囲か。オンプレ/クラウド、SLA、サポート範囲など}
```

例(機能骨子): 「問い合わせログの取り込み・LLM 分析・レポート出力を Web UI 上で完結できる。運用は顧客側、支援はチャットサポートのみ。」

### 研修・教育型: プログラム / 対象 / カリキュラム

顧客の人材のスキル・知識・行動を変えるサービス(研修・トレーニング・コーチング等)。典型入力パターン: 「研修」「トレーニング」「◯◯を教える」「◯人を育てる」等の語彙。受講対象・期間・演習形式への言及があるかも判定材料。

サブ構造:

```markdown
### プログラム: 全体設計と学習目標
{何が学べて、終わったときにできるようになることは何か}

### 対象: 誰向けか
{受講対象者の役割・レベル・前提知識}

### カリキュラム: 内容と進め方
{セッション構成・演習形式・期間・成果確認方法}
```

例(プログラム骨子): 「コールセンターマネージャー向けに、問い合わせデータを LLM で分析して打ち手を作る一連の流れを、実データを使って 2 日間で体験・習得する。」

### その他: 骨子「何を扱う / どう関わる / 何が起きる」

上記 4 型に当てはまらないサービスは、次の 3 要素を H3 で自由に構成する。項目名も自由に決めてよい。

```markdown
### 何を扱う
{サービスが対象とする素材・領域・状況}

### どう関わる
{顧客との関わり方・関与のスタイル・時間軸}

### 何が起きる
{関与の結果として発生する事象・変化・アウトプット}
```

## オプション章カタログ

必須 5 章以外に追加できる章。ヒアリング結果に以下の要素があるときのみ Phase 3 で追加を提案する(勝手に追加しない)。

| 章 | 追加を提案する条件 |
| --- | --- |
| 背景・なぜ今 | 元ネタやヒアリングで「Why now(市場や技術の変化)」「なぜ我々がやるのか(独自資産・原体験)」が明確に語られており、それを外すと読み手が納得しない場合 |
| 事例 | 顧客名または匿名で開示可能な実施実績が既にある(顧客名の可否をヒアリングで確認) |
| データの取り扱い | 顧客データを預かるサービスで、セキュリティ・保管期間・利用範囲への懸念が採用の障害になりうると判断される場合 |
| 提供体制 | ヒアリングでメンバー構成・専門性・支援体制が「差別化の根拠」として言及された場合 |
| 進め方 | 契約から納品までの流れが特殊(段階契約・PoC 前提・特殊な準備が必要等)で、口頭説明では不安が残る場合 |
| 料金 | 料金体系が確定しており、この文書で開示する方針(公開資料/顧客限定)が明確な場合 |

## セルフレビュー観点(7 項目)

Phase 5(および改訂モード)では、以下 7 観点すべてをドラフト全体に対して点検し、問題は Edit で修正してから「観点 / 判定 / 修正内容」を報告する。分量には上限を設けない(元ネタから引き出せた内容を制約で削らない)ため分量チェックはしない。代わりに冗長・重複・根拠のない水増しは観点 4・5 で弾く。

1. アウトカムが「顧客の業務・意思決定・数値・リスクの変化」の言葉で書けているか。納品物の言い換え(「レポートを納品します」)で止まっていないか。少なくとも 1 側面は具体的に書かれているか
2. プロセス / アプローチ / 機能等の中身章から差別化が読み取れるか。単なる手順・機能の羅列になっていないか
3. 課題 → 中身 → アウトカムの連鎖が破綻していないか(解かれない課題、アウトカムに繋がらない中身、が残っていないか)
4. 顧客視点の言葉になっているか(自社機能の自慢でなく、顧客の課題・利益の言葉か)。同じ主張の言い換え・冗長な繰り返しがないか
5. 数値・実績・顧客名はすべて根拠(ユーザー提供の事実)があるか。根拠のないものは漏れなく TBD になっているか。根拠のない形容詞・水増し表現で分量を稼いでいないか
6. 本文中の TBD と末尾 TBD 一覧が 1:1 で対応しているか
7. 各セクションが独立して読んで意味が通るか(NotebookLM / Claude Design がセクション単位で拾い読みしても骨子が伝わるか)。専門用語・社内用語が初見の読み手に通じる言葉に置き換わっているか
